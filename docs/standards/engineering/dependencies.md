# Dependency and Supply Chain Security

Policy for managing dependencies, vendored code, and supply chain integrity across the human project.

**Cross-references:** [../security/threat-model.md](../security/threat-model.md), [performance.md](performance.md)

---

## 1. Philosophy

human's core value proposition is **zero dependencies beyond libc**. Every dependency is a liability: binary size, attack surface, update burden, license risk. The default answer to "should we add a dependency?" is **no**.

## 2. Dependency Tiers

| Tier                              | Examples                     | Policy                                                                          | Approval                     |
| --------------------------------- | ---------------------------- | ------------------------------------------------------------------------------- | ---------------------------- |
| **Tier 0 — libc only**            | POSIX headers, C11 stdlib    | Always allowed                                                                  | None                         |
| **Tier 1 — Optional C libraries** | SQLite, libcurl              | Compile-flag gated (`HU_ENABLE_*`), must not be required for core functionality | Documented in CMakeLists.txt |
| **Tier 2 — Build tools**          | CMake, clang, zig            | Required for builds but not shipped                                             | Version pinned in CI         |
| **Tier 3 — UI/Web dependencies**  | npm packages (ui/, website/) | `package-lock.json` committed, `npm audit` in CI, review on update              | PR review                    |
| **Tier 4 — Dev/test only**        | vitest, playwright, prettier | Never shipped, version pinned                                                   | Standard update              |
| **Tier 5 — Scripts**              | Python scripts (scripts/)    | stdlib only preferred, pip packages pinned if used                              | PR review                    |

## 3. Adding a New Dependency

Before adding any dependency:

1. **Search existing code** — is the functionality already implemented or trivially implementable?
2. **Measure the cost** — binary size impact, transitive dependency count, license
3. **Evaluate alternatives** — can we vendor a single file instead of adding a package?
4. **Document the justification** — commit message must explain why the dependency is necessary

### Decision Checklist

- [ ] No existing implementation covers this need
- [ ] Binary size impact measured and acceptable (< 50 KB for C, < 100 KB for JS)
- [ ] License is MIT, BSD-2, BSD-3, ISC, Apache-2.0, or public domain
- [ ] Transitive dependency count is acceptable (< 5 for npm packages)
- [ ] Package has active maintenance (commit in last 6 months)
- [ ] No known CVEs at time of addition

## 4. Version Pinning

| Surface           | Mechanism                                          | Update Cadence                                   |
| ----------------- | -------------------------------------------------- | ------------------------------------------------ |
| C libraries       | System package manager or vendored                 | On security advisory                             |
| npm (ui/)         | `package-lock.json`                                | Monthly `npm audit fix`, quarterly major updates |
| npm (website/)    | `package-lock.json`                                | Monthly `npm audit fix`, quarterly major updates |
| Python (scripts/) | `requirements.txt` or inline pins                  | On security advisory                             |
| GitHub Actions    | SHA pinning (`uses: actions/checkout@v4` with SHA) | Quarterly                                        |

## 5. Vulnerability Scanning

| Check           | Tool                       | Cadence        | Gate                       |
| --------------- | -------------------------- | -------------- | -------------------------- |
| npm audit       | `npm audit`                | Every CI run   | High/critical = block      |
| SBOM generation | `security.yml` workflow    | Every release  | Generated, not blocking    |
| C library CVEs  | Manual review + Dependabot | On advisory    | Critical = immediate patch |
| License scan    | Manual review on addition  | Per-dependency | Non-compatible = block     |

## 6. SBOM

Software Bill of Materials is generated by the `security.yml` CI workflow on every release. Format: CycloneDX JSON. Includes all compiled C sources and linked libraries.

## 7. Vendoring Policy

When vendoring is preferred over a package dependency:

- Single-file libraries (e.g., stb_image.h, miniz.h)
- Must include: original license, version/commit hash, upstream URL
- Vendored files live in `vendor/` or inline in `src/core/`
- Update only on security advisory or needed feature
